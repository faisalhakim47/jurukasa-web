# Schema Git Diff Major Changes

This document summarizes the major changes currently shown by `git diff -- web/schemas`.

At the time of writing, the schema diff touches 16 files with 3596 insertions and 2164 deletions. The overall direction is not a small patch. It is a schema-layer refactor that tightens accounting invariants, makes workflow ownership more explicit at the database level, and consolidates previously separate features.

## Scope of the Diff

Changed migration files:

- `001-accounting.sql`
- `002-pos.sql`
- `003-chart-of-accounts.sql`
- `004-revenue-tracking.sql`
- `005-fixed-assets.sql`
- `006-account-reconciliation.sql`
- `007-cash-count.sql` removed entirely

Changed test files:

- `001-accounting-basic.test.js`
- `001-accounting-fiscal-year-closing.test.js`
- `001-accounting-fiscal-year-reversal.test.js`
- `001-accounting-reporting.test.js`
- `002-pos.test.js`
- `004-revenue-tracking.test.js`
- `005-fixed-assets.test.js`
- `006-account-reconciliation.test.js`

The test changes are important because they show intent. Many of the new rules are enforced by triggers, so the tests are the clearest proof of the expected workflow after the refactor.

## High-Level Themes

The diff introduces five major architectural shifts:

1. Journal entries are becoming more explicit, more immutable, and more tightly linked to higher-level business workflows.
2. Cash flow reporting moves to a direct-method, line-classified model instead of relying on broad account tags alone.
3. POS, fixed assets, and reconciliation now treat posted records as database-owned facts with much stricter mutation rules.
4. Fiscal year close and reversal workflows now require explicit journal entry references for generated accounting events.
5. Physical cash count is no longer a separate migration. It has been folded into a unified reconciliation checkpoint model.

## 1. Accounting Core Refactor in `001-accounting.sql`

### 1.1 Config and naming cleanup

The `config` table changes `description` to `note`, and the default seed data is updated accordingly.

This is a naming alignment change, but it also reflects a wider pattern in the diff: free-text metadata is being normalized around `note` instead of mixing `description`, `reference`, and other similar names.

### 1.2 Better timestamp semantics on account hierarchy maintenance

The child-account delete trigger now updates parent `update_time` in milliseconds using:

- `CAST(strftime('%s', 'now') AS INTEGER) * 1000`

This fixes a unit mismatch. The rest of the schema uses Unix epoch milliseconds, so the previous seconds-based update was inconsistent.

### 1.3 Cash flow tagging model is simplified and rebuilt

Several old cash flow tags are removed from `account_tags`, leaving only the account-level marker for cash equivalent accounts. In their place, the migration adds a dedicated `cashflow_categories` table with fixed categories such as:

- Customer Receipt
- Supplier Payment
- Rent Payment
- Reconciliation Adjustment
- Asset Purchase
- Loan Proceeds
- Capital Injection

This is a major design change. Instead of trying to infer the cash flow statement entirely from account tags, the schema now expects line-level classification on journal entry lines.

### 1.4 Journal entry IDs are no longer auto-generated by SQLite

`journal_entries.ref` changes from `INTEGER PRIMARY KEY AUTOINCREMENT` to `INTEGER PRIMARY KEY`.

This means the application or calling workflow must now provide the journal entry reference explicitly. The matching test changes confirm this direction: the tests stop relying on `RETURNING ref` from an auto-generated insert and instead create deterministic refs in userland.

Why this matters:

- Generated accounting workflows such as fiscal year close, reversal, depreciation, purchase posting, sale posting, stock taking, and reconciliation now need predictable references.
- Tests become deterministic without relying on SQLite auto-increment behavior.
- Higher-level workflows can reserve references before firing cascaded trigger logic.

### 1.5 Source metadata is removed from journal entry headers

The following `journal_entries` columns are removed:

- `source_type`
- `source_reference`
- `created_by`

The replacement pattern across the diff is stronger relational linkage instead of generic provenance fields. Later migrations add dedicated foreign key columns on `journal_entries` such as `purchase_id`, `sale_id`, `stock_taking_id`, `fixed_asset_id`, and `reconciliation_id`.

The design is shifting away from descriptive origin metadata toward concrete ownership links.

### 1.6 Posted journal entries are now substantially immutable

New triggers prevent mutation of posted journal entries except for the narrow case where reversal workflows fill `reversed_by_ref` on the original posted row.

This is a significant tightening of accounting integrity. After posting:

- `post_time` cannot change
- core fields like `entry_time`, `note`, `reversal_of_ref`, and `idempotent_key` cannot change
- only the allowed reversal bookkeeping path may update `reversed_by_ref`

The tests also add coverage to ensure lines cannot be added to posted journal entries.

### 1.7 Journal entry lines now carry cash flow classification

`journal_entry_lines` removes:

- `description`
- `reference`

and adds:

- `note`
- `cashflow_activity`
- `cashflow_category`

Validation rules enforce that:

- cash equivalent accounts must provide cash flow classification
- non-cash accounts must not provide cash flow classification
- activity and category must appear together
- the pair must exist in `cashflow_categories`

This is one of the most important changes in the whole diff. It moves cash flow reporting from loose inference to explicit transaction-line classification.

### 1.8 Fiscal year and reporting logic is adjusted to respect generated entries

The diff shows multiple trigger changes around fiscal year posting and reversal, plus an important reporting correction:

- the income statement view now excludes the closing journal entry so the view represents pre-closing activity even after the fiscal year has been closed

This prevents generated close entries from polluting operating-period reporting.

### 1.9 Cash flow statement generation is rebuilt

`cashflow_statement_lines` is renamed semantically from `line_description` to `line_note`, and a new `cashflow_report_generation_trigger` is introduced.

The comment in the migration is explicit: the cash flow statement is now auto-generated using the direct method by aggregating classified cash transactions from `journal_entry_lines`.

That is the core reporting shift of this migration.

## 2. POS Schema Hardening in `002-pos.sql`

### 2.1 Transaction price semantics are clarified

The header comments are rewritten to distinguish:

- `inventories.unit_price` as the current standard selling price per unit
- transaction-line `price` columns as the total contractual price for the quantity

This removes ambiguity between catalog pricing and factual transaction pricing.

### 2.2 Internal control flags are added

The migration introduces `internal_control_flags` with an `inventory_mutation_guard` seed row.

This suggests a stronger trigger-based protection scheme for inventory mutation side effects, especially when posting purchases, sales, and stock takings.

### 2.3 Inventories now have stronger protection around system-managed fields

New triggers prevent manual updates to system-managed inventory valuation fields.

This is consistent with the broader direction of the diff: balances and valuation fields should be derived by triggers, not edited directly by application code.

### 2.4 Supplier document ingestion is introduced

New tables are added for:

- `documents`
- `captures`
- `capture_dimensions`
- `capture_enhancements`
- `capture_recognitions`

This is a substantial new capability. It indicates the POS schema is starting to support collected supplier documents and structured capture metadata, likely as a basis for OCR or document-assisted purchase entry.

The tests reinforce this by adding coverage for purchases linked to collected supplier documents.

### 2.5 Supplier inventory becomes informational instead of conversion-based

The repository memory and the tests both point to the same design change:

- supplier inventory rows are now supplier-specific labels and references
- the earlier quantity-conversion model is removed from the workflow

The schema is moving toward storing the supplier-facing name and unit-of-measurement directly on purchase lines instead of treating supplier inventory as a conversion engine.

### 2.6 Business workflows now own the journal entry through one-to-one links

The migration adds dedicated nullable foreign keys on `journal_entries`:

- `purchase_id`
- `sale_id`
- `stock_taking_id`

Each gets a unique index so one journal entry can belong to at most one business record of that type.

This is a very important architectural change. Instead of storing an opaque source label on the journal entry, the database now expresses exact ownership.

### 2.7 Posted purchases, sales, and stock takings become much harder to mutate

The diff adds many new triggers that enforce:

- posting validation before `post_time` is accepted
- immutability of `post_time`
- immutability of posted accounting fields
- insert, update, and delete prevention on posted detail rows

This applies across:

- `purchases`
- `purchase_lines`
- `sales`
- `sale_lines`
- `sale_discounts`
- `sale_payments`
- `stock_takings`

The schema is drawing a harder line between draft records and posted records.

### 2.8 Sale line updates are intentionally almost forbidden

The comments around `sale_lines_prevent_update_trigger` make the intent clear:

- the only allowed update shape is the automatic cost initialization done by the insert-time cost trigger
- manual updates, including cost overrides, are blocked

This protects cost-of-goods-sold and inventory valuation from application-side drift.

### 2.9 POS inventory accounts are protected from manual journal misuse

New triggers such as `journal_entry_lines_prevent_manual_pos_inventory_update_trigger` strengthen the rule that inventory-affecting accounting entries for POS workflows must come from the owning domain workflow, not from arbitrary manual journal entry edits.

This trigger is important beyond POS itself because later migrations keep redefining it as new workflow link columns are added to `journal_entries`.

### 2.10 The tests show stricter transactional invariants

`002-pos.test.js` adds coverage for:

- negative quantity and price rejection
- required journal entry linkage before posting
- prevention of mutating posted purchase and sale records
- duplicate line prevention
- payment total validation against invoice amount
- discount and payment fee bounds
- stock taking snapshot validation
- protection against turning manual journal lines into POS inventory mutations

This means the schema is no longer assuming the application will behave. It is explicitly defending the invariants in SQLite.

## 3. Smaller Supporting Adjustments in `003-chart-of-accounts.sql` and `004-revenue-tracking.sql`

### 3.1 Chart of accounts migration is adjusted to fit the new base accounting rules

`003-chart-of-accounts.sql` changes are small compared with the other migrations. The diff suggests alignment work rather than a redesign.

In practice, this means the migration was updated to remain compatible with the new accounting core, especially around stricter trigger behavior and account tagging expectations.

### 3.2 Revenue tracking gets boundary fixes and rebuild semantics

`004-revenue-tracking.sql` and its tests show several reporting corrections:

- fiscal year revenue summary logic is rewritten
- some ordering and date-boundary behavior is adjusted
- the tests add coverage for excluding today from period comparison while still keeping today in sparklines
- the tests add coverage for rebuilding daily revenue from journal entries
- the tests add coverage for fiscal year boundaries that do not start at midnight

This is not just a cleanup. It indicates the reporting layer is being made more faithful to exact accounting-period boundaries and recoverable from journal entry truth.

## 4. Fixed Assets Workflow Expansion in `005-fixed-assets.sql`

### 4.1 Fixed assets gain their own internal mutation guard

The migration adds `fixed_asset_internal_flags` with a `mutation_guard` seed value.

This mirrors the POS pattern and signals that fixed asset side effects are also trigger-managed with re-entrancy or mutation-guard protections.

### 4.2 Fixed assets now link back to their owning journal entry

`journal_entries` gains `fixed_asset_id` with a unique index.

This extends the same ownership model seen in POS. A posted journal entry is increasingly treated as the accounting manifestation of exactly one domain event.

### 4.3 POS inventory protection triggers are widened to understand fixed assets

The migration drops and recreates the POS inventory protection triggers so they also allow fixed-asset-owned journal entries.

This is an important maintenance detail in the diff. The trigger is effectively a global allowlist of domain-owned journal entry links. Every time a new domain link is added to `journal_entries`, the trigger must be recreated with the new set of allowed foreign keys.

### 4.4 Fiscal year tables now track depreciation-generated entries explicitly

The migration adds:

- `fiscal_years.depreciation_journal_entry_ref`
- `fiscal_years.depreciation_reversal_journal_entry_ref`

This is a major functional change. Depreciation posting and depreciation reversal are no longer implicit side effects with anonymous entries. They are now first-class linked artifacts of the fiscal year workflow.

### 4.5 Fixed asset lifecycle rules are much stricter

The diff adds or tightens rules for:

- system-managed field protection
- creation and update validation
- deletion restrictions
- depreciation timing
- accumulated depreciation mutation control
- cleanup and reversal behavior during fiscal year workflows

The test file confirms the intent with many new scenarios, including:

- requiring a journal entry ref on asset creation
- enforcing fiscal year boundary behavior exactly at `begin_time` and `end_time`
- preventing multiple journal entries from linking to the same fixed asset
- requiring depreciation journal refs when close or reversal needs them
- preventing manual accumulated depreciation edits outside fiscal year workflows
- preventing deletion when posted acquisition history exists

### 4.6 Fixed asset acquisition now participates in cash flow classification

The tests add explicit coverage for acquisition payment classification as investing activity when the payment account is a cash equivalent account.

This ties the fixed asset migration directly into the new direct-method cash flow model introduced in `001-accounting.sql`.

## 5. Reconciliation Redesign in `006-account-reconciliation.sql`

### 5.1 The old multi-table reconciliation model is removed

The previous design included separate concepts for:

- reconciliation sessions
- statement items
- transaction matches
- discrepancies
- completion state
- summary and helper views built around that workflow

Most of that model is deleted.

### 5.2 A new checkpoint-based reconciliation model replaces it

The central table becomes `reconciliation_checkpoints` with fields for:

- `account_code`
- `type` with `STATEMENT` or `PHYSICAL`
- `checkpoint_time`
- `external_balance`
- `book_balance`
- `adjustment_journal_entry_ref`
- `note`
- `create_time`

This redesign is much simpler and much more outcome-oriented.

Instead of importing statement lines and matching them one by one, the new workflow records a checkpoint balance and lets the database compute the factual book balance as of that time. If there is a discrepancy, one adjustment entry is created automatically.

### 5.3 Statement reconciliation and physical cash count are now one feature

This is one of the biggest product-level changes in the diff.

The new `type` column makes both workflows live in the same reconciliation table:

- `STATEMENT` for bank or external statements
- `PHYSICAL` for counted cash or similar physical totals

This reduces duplication and eliminates the need for a separate cash-count migration.

### 5.4 Journal entries now link directly to reconciliation checkpoints

`journal_entries` gets `reconciliation_id` with a unique index, plus posted-entry protection on that link.

Again, this follows the same pattern seen elsewhere in the diff: generic provenance fields are being replaced by exact relational ownership.

### 5.5 Inventory-protection triggers are recreated again

Because reconciliation now owns some journal entries too, the migration drops and recreates the inventory-protection triggers once more so the allowlist includes reconciliation-linked entries.

This repeated trigger redefinition is not accidental. It is a direct consequence of moving to explicit foreign-key ownership on `journal_entries`.

### 5.6 Reconciliation outputs are simplified to single-entry adjustments

The new trigger flow does three things when a checkpoint is inserted:

- validates the checkpoint
- computes `book_balance` from posted entries as of `checkpoint_time`
- creates one posted adjustment entry if `external_balance` differs from `book_balance`

Balanced checkpoints keep `adjustment_journal_entry_ref` as `NULL`.

This is much simpler than the earlier session and matching model, and it fits the product goal described in the migration header.

### 5.7 A new reconciliation history view replaces the old reporting views

The old reconciliation summary, outstanding transaction, and account history view set is removed.

In its place, the diff adds `reconciliation_history`, which appears to focus on the recorded checkpoint, computed discrepancy, and resulting adjustment state.

### 5.8 The tests confirm the new operational model

`006-account-reconciliation.test.js` is effectively rewritten. The new coverage includes:

- balanced checkpoints with no adjustment entry
- physical shortages and overages
- statement shortages and overages
- non-posting-account rejection
- non-cash-account rejection for physical counts
- requirement for the proper reconciliation tag account when an adjustment is needed
- requirement that `adjustment_journal_entry_ref` stay present when a discrepancy exists
- history view behavior
- factual book balance calculation at `checkpoint_time`

This is not an incremental enhancement of the old reconciliation system. It is a replacement.

## 6. `007-cash-count.sql` Is Removed

The entire `007-cash-count.sql` migration and its companion test file are deleted.

That removal is explained by the `006-account-reconciliation.sql` redesign:

- cash count is no longer a separate feature layer
- physical counting is now represented by `reconciliation_checkpoints.type = 'PHYSICAL'`
- discrepancy handling is unified through the same adjustment-entry mechanism used by statement reconciliation

This is a true migration consolidation, not just a rename.

## 7. Test Suite Signals and What They Mean

The test changes reveal the practical contract expected by the schema after this refactor.

### 7.1 Explicit journal entry refs are now the norm

Multiple test files introduce deterministic helpers like `genJeRef()` and stop relying on auto-generated journal entry IDs.

This is a strong signal that all application code inserting into `journal_entries` must now supply `ref` explicitly.

### 7.2 Fiscal year close and reversal flows require explicit generated-entry wiring

Tests now require:

- `closing_journal_entry_ref` when a closing entry is needed
- `reversal_journal_entry_ref` when a reversal is needed
- `depreciation_journal_entry_ref` when depreciation posting is needed
- `depreciation_reversal_journal_entry_ref` when depreciation reversal is needed

This makes generated accounting artifacts auditable and non-optional when real postings occur.

### 7.3 Contra revenue and exact-period behavior receive new coverage

The closing tests now include contra revenue handling, backdated posting protection into closed fiscal years, and empty-close cases that must not create orphan entries.

This shows the fiscal year workflow is being hardened for real accounting edge cases.

### 7.4 Posted records are treated as immutable facts across domains

New tests repeatedly assert that posted or recorded rows cannot be mutated afterward. This pattern appears in:

- journal entries
- purchases
- sales
- stock takings
- fixed assets
- reconciliation checkpoints

That consistency is one of the clearest cross-cutting themes in the diff.

## 8. Practical Consequences for Future Development

Anyone extending the schema or writing application code against it should assume the following rules now hold.

### 8.1 Do not rely on SQLite to generate `journal_entries.ref`

Callers should allocate the reference themselves before insert.

### 8.2 Do not treat posted records as editable business objects

The database increasingly treats posting as the transition from draft intent to immutable accounting fact.

### 8.3 Cash flow reporting now depends on line-level classification

If a journal line touches a cash equivalent account, the caller must supply `cashflow_activity` and `cashflow_category` correctly.

### 8.4 Domain ownership of journal entries is now explicit

Purchases, sales, stock takings, fixed assets, and reconciliation checkpoints each use dedicated foreign key links on `journal_entries`.

This is better than generic source metadata, but it means future migrations must maintain those ownership constraints carefully.

### 8.5 Trigger maintenance got more coupled

The repeated drop-and-recreate pattern for `journal_entry_lines_prevent_manual_pos_inventory_trigger` is a maintenance warning.

If a future migration adds another journal-entry owner column, that trigger must be recreated again with the full updated allowlist. Missing one owner column would create false validation failures.

### 8.6 Reconciliation and cash counting should now be designed as one workflow

Any application or UI layer still assuming a separate cash count domain is now out of sync with the schema direction.

## 9. Summary

The major change in this diff is not any single trigger or table. It is the overall movement of business logic toward explicit database-owned workflows.

The schema is now doing more of the following itself:

- deciding which records are mutable and when
- linking domain events to their exact accounting entries
- requiring explicit references for generated journal entries
- classifying cash movement at the line level for direct-method reporting
- consolidating overlapping workflows into fewer, more opinionated models

In short, the schema is becoming stricter, more deterministic, and more domain-specific. The large accompanying test rewrite shows that this is intentional and foundational, not incidental cleanup.
